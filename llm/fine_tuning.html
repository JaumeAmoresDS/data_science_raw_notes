Scenarios

  • Open Source LLM + Fine-tuning can outperform foundational SOTA models (e.g., ChatGPT).

When to fine-tune

  • Two ways of learning:
    • Weights: pre-training, fine-tuning
    • Prompting, via RAG.
  • Analogy:
    • Fine-tuning is like studying one week in advance.
    • Prompting is like taking exam with open notes.
  • Not Good for:
    • Learning new knowledge. Can increase hallucinations.
  • Good for:
    • Specialized tasks, in similar ways to RAG.
      • With many examples

When to fine-tune, more analogies

  • Zero shot learning: describe task with words.
  • Few shot learning: give few examples of solving task in prompt (e.g., via RAG or manually)
  • Fine tuning: allow person to practice task.
    • In applications with concrete well-defined tasks where it is possible to collect a lot of data and “practice” on it.
  • Karpathy’s tweet

Fine-tuning vs others complete guide

Possibilities along two axis: - Complexity / Cost dimension (higher to lower): - From scratch training - Reinforcement Learning from Human Feedback - Fine Tuning, Retrieval Assisted Generation => Problems Addressed dimension: Form problems - Fine Tuning, Factual Problems - RAG - Static / Dynamic Example Selection - Manual / Automatic Prompt Tuning

  • Fine tuning is good at:
    • Learning the style or form of language.
    • Examples:
      • Pure autoregressive model on Q & A, since there’s plenty of Q&A data.
      • Pure autoregressive model on instruction following, since there’s plenty of data.
      • Imitate style, e.g., Shakespeare, since there is lot of training material. Also, legal jargon, claim responses
      • Imitate structure / format, like resumes.
  • It is not good at:
    • Learning new concepts that do not exist in the base knowledge of the foundational model.
    • Example:
      • Replace Romeo with Bob in a set of texts and fine-tune. See if it forgets association of Romeo with Juliet and learns that it is Bob who was with Juliet. Fails because the Romeo concept is ingrained in the base knowledge.
      • Answer questions like “who said to be or not to be” after fine-tuning on set of Shakespeare works.
        • Better use search engine (RAG) to retrieve those answers and inject them in prompt for LLM.
        • Question: what does LLM add here, if the search engine is already finding relevant answers? Remove FP?
  • Fine tuning is for form, not facts

Quantization, LoRA and qLoRA

  • Model quantization: fit model (e.g., 7B Llama) in memory.
  • LoRA: efficiently fine-tune model by reducing number of parameters to be trained
  • qLoRA: same, but deals with quantized models.

Fine Tuning OpenAI Models - Best Practices

  • Video

When to fine-tune

  • Good for:
    • Follow given format or tone.
    • Processing the input following specific, complex instructions.
    • Improve latency (shorter prompt to process, since we don’t need to include multiple examples)
    • Reducing token usage (again, shorter prompt)
  • Not good for:
    • Teaching new knowledge.
    • Performing multiple, unrelated tasks:
      • Do prompt-engineering or use multiple FT models.
    • Including up-to-date content in responses
      • Too much latency.
      • Use RAG instead.

Fine-tuning vs other optimization techniques

OpenAI blog post

  • Context optimization (what the model needs to know): move towards prompt / RAG
  • LLM optimization (how the model needs to act): move towards fine-tuning.
    • Example: say “I don’t know” more often when uncertain.
  • We can use both RAG and fine-tuning.

Preparing the dataset

  • Take the set of instructions and prompts that you found work best for the model prior to fine-tuning, and include them in every training example.
  • Take at least 50-100 examples.

Best Practices

Curate examples carefully

  • Better spend time on few examples of higher quality.
  • Consider prompt baking: using basic prompt to generate initial examples.
  • Failure Examples or cases with issues.
  • Consider Balance and diversity of data.
  • Multi-turn conversations: take representative examples.
  • Make sure messages contain all information needed in the response.

Iterate on hyper-parameters

  • Start with defaults.
  • If not converging, increase the learning rate multiplier.
  • If doesn’t seem to learn, increase epochs.
  • If becomes less diverse than expected, decrease epochs by 1-2.

Establish a baseline

  • Use zero shot or few shot learning as baseline.

Automate your feedback pipeline

  • Introduce automated evaluations for:
    • potential problem cases
    • clean up data and use as new training
  • Consider G-Eval approach (LLM as judge with GPT4) for automated testing with scorecard.

Optimize for latency and token efficiency

  • GPT3.5 fine-tuned for matching accuracy of GPT4 at lower cost / latency.
  • Reducing instructions with fine-tuned model.

Other uses for fine-tuning

  • improved function calling or output parsing
  • Caution: if the number of possible functions is too large it might be better to include the desired ones in the prompt.
